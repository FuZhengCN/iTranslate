# i18n: Chinese/English Bilingual UI

**Date:** 2026-05-22
**Status:** Design approved

## Summary

Add bilingual (zh_CN / en) support to all user-facing surfaces. Language auto-detects from `navigator.language` (`chrome.i18n.getUILanguage()`) — no manual override. App name: "通译" (Chinese) / "iTranslate" (English).

## Scope

| Surface | Mechanism |
|---------|-----------|
| `manifest.json` | `__MSG_*__` placeholders |
| `popup.html` | `__MSG_*__` placeholders |
| `popup.ts` | `chrome.i18n.getMessage()` via `t()` helper |
| `settings.html` | `__MSG_*__` placeholders |
| `settings.ts` | `chrome.i18n.getMessage()` via `t()` helper |
| Content script | No user-facing hardcoded strings currently — no changes |

## File Structure

```
_locales/
  en/messages.json
  zh_CN/messages.json
src/shared/
  i18n.ts                # t() wrapper, detectUILanguage()
```

## Helper Module (`src/shared/i18n.ts`)

```ts
export function detectUILanguage(): 'en' | 'zh_CN' {
  const lang = chrome.i18n.getUILanguage();
  if (lang.startsWith('zh')) return 'zh_CN';
  return 'en';
}

export function t(key: string, substitutions?: string | string[]): string {
  return chrome.i18n.getMessage(key, substitutions) || key;
}
```

`detectUILanguage()` is available but not strictly needed at runtime — Chrome's `chrome.i18n.getMessage()` already resolves from the correct locale automatically. It exists for potential conditional logic (e.g., CJK-specific rendering).

## Language Detection

- `chrome.i18n.getUILanguage()` returns the browser's UI language (e.g. `"zh-CN"`, `"en-US"`)
- `zh*` → `zh_CN`, everything else → `en` (default fallback)
- No user preference stored — always follows browser

## Integration

### manifest.json
- Add `"default_locale": "en"`
- Replace `name`, `description`, `action.default_title` with `__MSG_*__`

### HTML files
- All text content and `title` attributes use `__MSG_key__` placeholders
- Chrome processes these at extension page load time — no build step needed

### TypeScript files
- Import `{ t }` from `../shared/i18n`
- Replace all hardcoded English strings with `t('key')` calls
- For substitutions: `t('connectionError', [err.message])`

## Decisions

| Decision | Rationale |
|----------|-----------|
| Chrome i18n API, not custom module | Standard MV3 approach, handles manifest + HTML + JS uniformly |
| No manual language override | Simpler — no setting to persist, follow the browser |
| Language names stay native | Standard for language pickers, avoids confusion |
| Flat key naming (no prefixes) | Only ~25 keys, context clear from key name |

## Exclusions

- No changes to `LANGUAGE_OPTIONS` (stay native)
- No changes to background service worker (no UI strings)
- No changes to CSS
- No changes to content script (no hardcoded English in DOM)
- No `_locales` for other languages (ja, ko, etc.) — English is the fallback for all non-Chinese locales
