## 7.1 通用对接：适配全平台 OpenAI 兼容 API

iTranslate 翻译引擎的核心设计思想是**模型无关、服务商无关**。项目不硬编码绑定任意 AI 厂商接口，只要服务遵循 OpenAI `/chat/completions` 统一调用规范，DeepSeek、OpenAI、Claude 兼容接口、本地 Ollama 等各类大模型服务，均可无缝接入、即插即用。

这套高兼容性的核心，是一套标准化、可配置的统一请求体。引擎核心请求逻辑封装在 `translator.ts` 文件中，所有翻译请求均遵循统一参数结构，完整代码与对应工程设计意图如下：

```typescript
fetch(`${endpoint}/chat/completions`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    Authorization: `Bearer ${settings.apiKey}`,
  },
  body: JSON.stringify({
    model: settings.model,           // 动态读取用户配置模型
    messages: [
      { role: 'system', content: settings.systemPrompt },
      { role: 'user', content: prompt },
    ],
    temperature: 0.1,
    max_tokens: maxTokens,           // 动态估算最大生成长度
    thinking: { type: 'disabled' },  // 适配 DeepSeek 推理模型
  }),
});
```

本文所有请求参数均经过大量实战迭代与权衡，每一项配置都具备明确的工程目的，下面逐一拆解核心设计逻辑：

### 7.1.1 可切换模型：model 与 messages

`model` 与 `messages` 是 OpenAI 兼容接口的标准核心字段。本项目将模型名称、接口地址、API 密钥全部交由用户配置、动态读取，全程无需硬编码。用户可随时切换 `deepseek-v4-flash`、`gpt-4o` 等任意兼容模型，**无需修改代码、无需重新打包**，真正实现模型自由切换、服务商无感替换。

### 7.1.2 最优随机性：temperature=0.1

温度参数是翻译场景的核心调参，取值绝非随意设定。网页翻译的核心诉求是**语义绝对忠实、语句自然流畅**，温度的取值需要在“精准度”与“自然度”之间做精准平衡。

若设置 `temperature=0`，模型将进入完全确定性输出状态，无任何随机空间，生成的译文容易生硬僵化、丢失自然语言语感；若温度取值过高，模型会过度自由创作，极易出现语义偏移、增译、漏译、主观改写等问题，无法满足翻译的严谨性要求。

经过反复实测验证，`0.1` 是该场景下的最优取值：保留极微弱的随机空间，保障译文语句通顺自然、贴合人类阅读习惯，同时严格约束模型不偏离原文语义，完美适配网页精准翻译的核心需求。

### 7.1.3 动态 Token 上限：max_tokens

项目摒弃固定的 `max_tokens` 配置，通过 `estimateTokens()` 函数实现动态 Token 上限估算，精准适配不同长度、不同语种的文本内容。核心估算逻辑为：Prompt 字符数 × 0.6（输出字符比例）× 1.5（Token 字符换算系数），同时将结果约束在 `[512, 8192]` 的合理区间，并预留 256 Token 缓冲余量，规避边界异常。

这套轻量化、低损耗的估算算法无需复杂模型计算，即可解决两大核心问题：有效避免 Token 不足导致的译文截断问题，同时杜绝固定超大 Token 值带来的资源浪费，兼顾翻译完整性与使用经济性。

### 7.1.4 推理模型兜底：thinking 禁用策略

`thinking: { type: 'disabled' }` 是专门适配 DeepSeek 推理模型（V4 系列模型均内置 thinking 模式，V3 时代则体现在 deepseek-reasoner 等推理模型上）的关键避坑参数。

推理类模型会在生成正式译文前，优先执行内部逻辑推导，产生大量推理 Token。这类 Token 会被正常计入 `completion_tokens` 消耗，但不会展示在接口返回的 `content` 字段中。若未手动禁用该模式，会出现典型的诡异问题：接口调用成功、Token 正常扣费、无任何报错，但最终页面无翻译内容。本章后续将结合真实实战日志，完整复盘问题成因、排查过程与落地优化方案。

### 7.1.5 去中心化调用：端点与密钥私有化

接口地址 `endpoint` 与用户密钥 `apiKey` 均由用户自主配置、本地安全存储。所有 AI 翻译请求直接从用户浏览器发起，无需经过项目中间服务器转发、中转。

该架构设计的核心优势是**从根源规避隐私风险**：项目无任何数据采集、无文本内容转发行为，用户网页文本仅与用户指定的 AI 服务商直接交互，无需复杂的隐私合规声明，大幅提升用户使用信任感与产品合规性。

## 7.2 分层 Prompt 设计：标准化可量产翻译能力

Prompt 质量直接决定翻译效果上限。适用于程序化自动化翻译的优质 Prompt，绝非简单的“你是翻译助手”这类模糊指令。iTranslate 创新性采用**System 全局约束 + User 结构化入参**的双层 Prompt 体系，搭配解析层容错兜底机制，实现稳定、精准、可量产、可复用的自动化翻译能力。

### 7.2.1 System Prompt：全局翻译规则约束

项目默认的标准化系统提示词简洁精炼、无冗余话术，每一句指令都对应明确的工程约束，精准适配程序化调用场景：

```plain
You are a professional English-to-Chinese translator. Translate the following
text accurately while preserving the original meaning, tone, and formatting.
Only output the Chinese translation, nothing else.
```

分层拆解这套系统提示词的核心设计意图，每一条规则都具备极强的工程针对性：

1. **角色定位：professional translator**：精准定义模型身份为专业翻译人员，而非通用闲聊助手、文案创作工具。大模型会根据角色设定自动匹配严谨的措辞风格、输出规范，从源头规避口语化、随意化的无效输出。
2. **三维翻译准则：meaning、tone、formatting**：明确三大核心翻译标准——语义忠实原文、语气完全匹配、格式完整保留。杜绝摘要式翻译、改写式优化、排版错乱等问题，完整复刻原文的分段结构、排版样式与情绪风格。
3. **裸输出约束：Only output the Chinese translation**：这是程序化 AI 调用的**核心铁律**。严格禁止模型输出解释、寒暄、备注等多余内容，彻底规避冗余文本导致的程序解析失败、格式错乱问题，保障接口返回结果可直接结构化解析与渲染。

### 7.2.2 User Prompt：结构化编号入参

单网页通常包含数十段独立、零散的文本内容，批量翻译的最大技术难点是**保证原文与译文精准一一对应、互不错乱**。项目通过 `buildPrompt()` 函数实现结构化批量入参，依托标准化编号体系锁定文本映射关系，从Prompt层规避匹配异常：

```typescript
function buildPrompt(texts: string[]): string {
  const segments = texts.map((t, i) => `[${i}] ${t}`).join('\n\n');
  return `Translate the following texts. Each text is wrapped with a numbered tag.
Output the translations using the same numbered tags, one per line, with no
additional commentary:\n\n${segments}`;
}
```

### 7.2.3 优选 [N] 编号体系的核心原因

项目舍弃行业常用的 `1.`、`1)`、`1、` 等常规编号格式，统一采用`[N]` 方括号编号，核心目的是**彻底规避网页正文内容的干扰冲突**。

网页文章、技术文档、博客内容中，普遍存在“1. 引言”“2. 总结”等自带数字编号的文本。若采用普通数字编号，模型无法区分「指令编号」和「原文自带编号」，极易引发格式错乱、文本匹配错位、段落丢失等问题。而 `[N]` 格式在常规网页文本中出现概率极低，几乎无冲突风险，是批量翻译场景下稳定性最高的编号方案。

### 7.2.4 解析层容错：适配模型“不标准输出”

所有大模型都存在输出不确定性，即便 Prompt 规则严格约束，不同厂商、不同版本的模型仍可能输出非标准编号格式。因此项目坚守核心工程原则：**Prompt 是理想输出标准，解析层是落地兜底保障**，绝不默认模型会100%遵循指令。

`parseResponse()` 采用双优先级正则匹配策略，兼容主流输出格式，最大化解析成功率：

```typescript
const patterns = [
  /^\[(\d+)\]\s*(.+)/,       // 优先匹配标准 [0] 格式
  /^(\d+)[\.\)、]\s*(.+)/,   // 兼容 0. / 0) / 0、 等变体格式
];
```

解析逻辑采用分层兼容策略：优先精准匹配项目规定的`[N]` 标准格式，匹配失败后自动兼容各类主流编号变体。若所有规则匹配失效，程序会自动填充友好占位文案，保证页面功能不崩溃、用户可直观感知异常，实现优雅降级，保障使用体验。

### 7.2.5 可自定义 Prompt：产品核心差异化

区别于 Google 翻译、DeepL、Edge 内置翻译等功能固化的工具，iTranslate 开放了自定义 System Prompt 能力。用户可在扩展设置面板自由编辑默认翻译规则，按需适配各类阅读场景，实现个性化翻译效果：

- 日常阅读：修改 Prompt 要求译文更口语化、通俗易懂；
- 技术文档：新增规则保留所有专业术语英文原文；
- 文学阅读：保留原文修辞手法、文学质感与情绪氛围。

传统翻译工具仅能提供固定、统一的翻译结果，而 iTranslate 交付的是**可自定义、可个性化适配的翻译能力**，这也是 iTranslate 区别于市面通用翻译工具的核心能力。

## 7.3 批量并发策略：平衡速度、质量与成本

单网页往往包含数十至上百段零散文本，纯串行逐段翻译效率极低、体验卡顿，一次性全量批量翻译又极易触发 Token 超限、接口报错等问题。为平衡翻译速度、输出质量与调用成本，项目落地了一套**语言感知动态分批 + 有限并发 + 有序兜底 + 克制重试**的完整工程解决方案。

### 7.3.1 语言感知动态分批

批量翻译的核心拆分依据是 Token 数量，而非简单字符数量。中文、日文等 CJK 字符与英文等拉丁字符的 Token 消耗差距最高可达4倍，统一字符拆分规则会导致批次负载严重失衡，出现部分批次超限报错、部分批次资源浪费的问题。

项目通过 `segmentTokenEstimate()` 实现智能语种感知估算：采样文本前40个字符，判断 CJK 字符占比。占比超30%则按1.5 Token/字符估算，否则按0.35 Token/字符估算，将每批文本 Token 稳定控制在1500 Token左右。

该阈值经过大量实战权衡打磨：批次过小会割裂文本上下文，降低翻译一致性与准确率；批次过大会增加单请求延迟、提升接口报错概率，1500 Token 是兼顾翻译质量与响应速度的最优平衡点。

### 7.3.2 有限并发控制（最大3并发）

为避免并发过高触发 AI 服务商接口限流、IP 封禁问题，项目采用队列消费模式，固定开启3个工作线程并行处理分批任务。通过 `runNext()` 递归消费队列，当前批次执行完毕后自动拉取下一批任务，直至队列全部清空，在大幅提升翻译速度的同时，严格控制请求频率、保障接口稳定性。

### 7.3.3 克制精准的重试机制

项目摒弃行业常见的无脑全员重试逻辑，精准区分异常类型，仅对**可恢复异常**执行重试操作，最大化节省资源、提升稳定性：

- **支持重试场景**：429 请求限流、5xx 服务端异常，最多重试3次，采用 `2^attempt * 1000ms` 指数退避策略，为服务端预留充足的恢复与降温时间；
- **禁止重试场景**：4xx 客户端错误（401密钥错误、400参数错误、403权限不足等），直接抛出异常终止请求，避免无效消耗用户额度、浪费接口资源。

### 7.3.4 并发乱序兜底：位置映射排序

多批次并发请求的网络返回顺序完全随机，慢速批次、延迟批次可能滞后返回，直接拼接结果会造成译文段落错乱、匹配错位。项目通过 `origIndices` 数组精准记录每一批文本的原始位置，接口结果返回后，严格按照原始索引回填译文，**100%保障输出顺序与输入顺序完全一致**，彻底解决并发乱序问题。

### 7.3.5 推理模型空内容兜底

针对 DeepSeek 推理模型专属异常问题，项目新增专属兜底检测逻辑：当接口返回 `content` 为空、但`completion_tokens` 存在正常消耗时，判定为模型推理内容占满 Token、译文生成失败，自动纳入重试机制，实现用户无感知的故障修复，大幅提升模型兼容性。

## 7.4 缓存层设计：降本提速的核心基建

重复翻译相同网页文本，会造成不必要的 Token 损耗与网络延迟，影响使用体验、增加使用成本。因此，高性能、高可靠的缓存层是翻译类产品的核心基建。iTranslate 基于 IndexedDB 实现浏览器端持久化缓存，完美适配网页高频重复翻译场景。

### 7.4.1 存储选型：IndexedDB 最优解

对比浏览器主流的三种本地存储方案，IndexedDB 是翻译缓存场景的最优方案，各方案优劣对比如下：

- **localStorage**：仅5MB极小容量上限，网页批量翻译会快速溢出，完全无法适配海量文本缓存场景；
- **chrome.storage**：浏览器扩展专属存储，存在严格的读写频率限制，无法支撑批量翻译的高频并行读写需求，性能瓶颈明显；
- **IndexedDB**：大容量无硬性上限、支持事务操作、异步非阻塞读写、并行读写性能优异、稳定性强，是网页端海量文本缓存的标准最优方案。

### 7.4.2 缓存 Key 四重防坑设计

缓存 Key 的设计直接决定缓存命中率与准确性，稍有疏漏就会出现缓存错乱、内容覆盖、哈希碰撞、语种匹配错误等隐性 Bug。项目通过四重防护机制，彻底规避各类缓存异常问题：

1. **哈希+长度双校验**：采用 djb2 哈希算法生成文本唯一指纹，搭配文本长度作为双重校验维度，极致降低哈希碰撞概率；
2. **绑定目标语言维度**：缓存 Key 强制携带目标翻译语言参数，彻底修复切换语种后展示旧缓存内容的经典隐性 Bug；
3. **业务前缀隔离**：区分 `dict_` 词典查询缓存与 `seg_` 段落翻译缓存，不同业务缓存相互隔离、互不覆盖，杜绝业务数据错乱；
4. **原文二次强校验**：缓存实体完整存储原始文本，命中缓存时强制比对原文内容，即便出现极低概率的哈希碰撞，也能精准拦截错误缓存，保障准确性。

### 7.4.3 高性能读写策略

项目缓存查询采用 `cacheGetBulk()` 批量并行读取方案，依托 IndexedDB 并行事务能力，相比传统串行循环读取方式，性能提升数倍，适配大批量文本快速缓存匹配场景。

整体业务流程严格遵循**缓存优先**原则：触发翻译后，程序优先批量查询本地缓存，命中的文本直接复用本地译文、无需请求接口；仅未命中的新增文本，才会发起 AI 翻译请求，翻译完成后自动批量写入缓存。用户二次访问同一页面可实现零请求、零延迟、零成本翻译，极大优化体验、降低使用成本。

## 7.5 本章核心实战技巧

1. **优先适配多模型通用规则，而非单模型微调**：无需针对单一模型反复微调 Prompt，不同模型输出习惯差异极大。最优方案是设计通用精简 Prompt，在解析层做格式兼容，一套规则适配全模型，大幅降低维护成本；
2. **温度参数拒绝极端取值**：翻译场景温度不设0、不设高值，0.1是兼顾语义忠实度与语句自然度的黄金取值，完美适配网页翻译刚需；
3. **Token 估算必须区分语种差异**：中日韩 CJK 文本与拉丁文本 Token 消耗差距悬殊，通过语种感知采样估算，让每批任务负载更均匀，避免接口过载或资源浪费；
4. **多语言产品缓存必须绑定语种维度**：缓存 Key 遗漏语种是隐蔽性极强的高频 Bug，问题排查难度大，必须在设计阶段提前规避；
5. **推理模型配置双保险兜底**：使用 DeepSeek 推理模型时，需同时开启「禁用 thinking 模式」与「空内容检测自动重试」双机制，彻底解决模型扣费无输出的隐性故障。

- **双层标准化 Prompt 体系**：System Prompt 统一翻译规则与裸输出标准，结构化 User Prompt 实现批量文本有序入参，搭配解析层容错兼容，完美适配多模型差异化输出；
- **场景化最优参数策略**：`temperature=0.1` 精准平衡翻译忠实度与自然度，动态 Token 估算适配不同文本语种与长度，彻底解决译文截断与资源浪费问题；
- **高稳定批量并发机制**：通过语种感知动态分批、3并发队列限流、指数退避重试、原始位置映射排序，同时兼顾翻译速度、输出质量与接口稳定性；
- **高安全可靠缓存基建**：基于 IndexedDB 构建持久化缓存，四重 Key 防护机制杜绝错乱与碰撞，批量并行读写大幅提升响应速度，有效降低用户使用成本；
- **推理模型专属兜底方案**：双保险机制禁用模型思考模式、自动修复空输出异常，彻底解决 DeepSeek 推理模型的隐性适配问题。
