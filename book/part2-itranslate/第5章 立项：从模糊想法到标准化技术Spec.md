# 第5章 立项：从模糊想法到标准化技术Spec

**本章目标**：读完本章，你将完整掌握AI编程项目的标准化立项全流程：从一个随口冒出的模糊产品想法，依靠结构化需求梳理能力，收敛出边界清晰、决策明确、可直接指导编码的技术Spec。你将深度理解 brainstorming 技能的实战价值，并彻底吃透AI编程的一条核心铁律：先写Spec，再写代码。

## 5.1 项目缘起：从竞品体验到实战定位

2026年5月，我体验了市面上主流的Chrome翻译扩展「沉浸式翻译」。它的双语对照、划词翻译、多引擎切换等能力，搭配流畅稳定的交互体验，足以满足绝大多数用户的网页翻译需求。与此同时，我收听了其创始人Owen的访谈播客，了解到这款产品从个人业余项目，一步步迭代为百万用户级产品的完整历程。

深度体验后，我产生了一个明确的实战想法：这类轻量化浏览器翻译工具，功能模型清晰、技术栈标准，非常适合作为AI编程的完整实战案例。我可以基于它从零跑通「单人开发者 + Claude Code + DeepSeek」的全链路开发流程，真实验证AI辅助开发的落地能力与效率上限。

这里需要厘清核心初衷：本项目并非要复刻甚至超越成熟商业产品。在启动时，我几乎没有Chrome扩展开发经验，不具备与商业化工具比拼完整度、稳定性和生态的条件。这个项目的核心价值，在于**选择一个难度适中、边界可控、场景真实的Side Project，用来沉淀可复用的AI工程化方法论**。

翻译扩展恰好满足所有实战条件：核心功能收敛清晰（页面翻译、划词翻译、词典查询）、技术栈标准统一（Manifest V3 + TypeScript）、有成熟竞品可参考交互范式，非常适合从零落地、完整复盘AI开发流程。确定方向后，我立刻借助Claude Code启动项目立项与需求梳理工作。

## 5.2 实战复盘：Brainstorming 需求澄清

AI编程最大的误区，就是拿到一句模糊需求就直接开始写代码。在真实项目中，开发最大的风险从来不是“功能写不出来”，而是“埋头开发很久，最后发现需求跑偏、范围失控、产品不对位”。

因此，AI项目立项的第一步，绝对不是编码，而是**需求澄清**。我直接使用Claude Code的 `brainstorming` 工作流，让AI帮我结构化查漏补缺、收敛模糊想法。

我向Claude Code输入的初始需求如下：

```plain
[我的输入]
> 我想做一个 Chrome 翻译扩展，类似沉浸式翻译的双语对照效果。
> 翻译引擎用 DeepSeek API。帮我用 Claude Code 从零搭建这个项目。
```

Claude Code 并没有直接生成代码，而是按照标准化 brainstorming 流程，反向补齐需求盲区，抛出了五个决定产品形态的核心问题，帮我把模糊想法逼成可落地的决策：

```plain
[Claude Code 追问]
> 在开始之前，我想先理清几个核心问题：
>
> 1. 目标用户是谁？习惯自己配 API Key 的开发者，还是期望开箱即用的普通用户？
> 2. 核心差异化是什么？市场上翻译扩展已经很多，你的扩展解决了什么细分问题？
> 3. 翻译方向：只做英→中，还是支持多语种互译？
> 4. 触发方式：用户手动点击翻译，还是打开页面自动翻译？
> 5. 用户自带 Key 意味着需要 Settings 页面，这个页面你打算放哪些配置项？
```

这些问题看似基础，实则是项目的“定位锚点”。前置厘清这些问题，和写完代码再发现偏差，开发成本天差地别。

以「目标用户」为例：如果面向普通用户，产品必须做到零配置、自动翻译、内置免费引擎，降低使用门槛；如果面向开发者用户，则可以接受手动配置、自定义模型、自定义Prompt，主打可控性与高质量翻译。

我最终敲定核心定位：**产品主打开发者群体，面向愿意自主配置 API Key、追求翻译质量可控、Prompt 可自定义的技术用户**。

仅此一个决策，直接锁定了后续所有产品设计：无需内置免费翻译引擎、默认手动触发翻译、优先配置Settings页面、支持切换DeepSeek等多款模型。

**立项收敛结论**：本项目是一款**面向开发者的轻量化Chrome双语翻译扩展**，支持用户自主配置API密钥、自定义翻译Prompt、手动触发翻译，V1版本优先落地英译中核心能力，全程基于DeepSeek模型提供翻译服务。自此，所有架构、功能、技术选型都有了统一的判断标准。

## 5.3 技术选型：基于约束做最优决策

产品定位与需求明确后，即可进入技术选型阶段。Chrome扩展开发拥有专属的规范约束与生态边界，技术选型不能仅凭个人喜好，必须结合官方规范、长期兼容、开发效率、审核规则、用户隐私等多维度权衡。本节将从核心清单规范、构建工具、权限策略三大核心维度，拆解本项目的全套选型逻辑。

### 5.3.1 清单规范：选择 Manifest V3

Chrome扩展存在两大主流清单规范：Manifest V2 与 Manifest V3。目前Google已明确V2版本的淘汰时间线，新版Chrome已逐步停止对V2的支持，所有新项目都必须基于V3开发，选择V2只会埋下后续重构的隐患。

Manifest V3 最核心的变革，是将传统常驻的Background页面，替换为**可休眠、可唤醒的Service Worker**。这一设计提升了浏览器性能与安全性，但也带来了明确的开发约束：后台无法维护全局持久状态、不支持长时间定时器、进程休眠后临时变量会全部丢失。

该约束直接引发经典的「冷启动竞态问题」：当Service Worker处于休眠状态时，前端发送的消息可能早于监听器注册完成，导致出现 `Receiving end does not exist` 报错。为解决该问题，项目最终设计了**3次重试机制（间隔600ms）**，为后台进程预留充足的唤醒与注册时间，彻底规避通信异常问题。

整体来看，尽管V3存在一定开发门槛，但它是Chrome官方唯一长期主推的规范，适配商店审核、浏览器兼容、后续迭代，是新项目的唯一最优解。

### 5.3.2 构建工具：Vite + crxjs 组合方案

当前Chrome扩展开发主要有三种构建方案，各有优劣。我借助Claude Code完成多维度对比分析，最终结合开发效率与长期维护性敲定最优方案。

| 构建方案                  | 核心优点                                          | 核心缺点                                         |
| :------------------------ | :------------------------------------------------ | :----------------------------------------------- |
| 手动配置（tsc + 脚本）    | 零第三方依赖、项目完全自主可控                    | 需手动维护Manifest配置、无热更新、迭代效率极低   |
| Webpack + 插件            | 生态成熟、插件体系完善、兼容性强                  | 配置繁琐冗余、构建速度慢，2026年新项目无适配必要 |
| Vite + @crxjs/vite-plugin | 极速HMR热更新、自动同步Manifest版本、开发体验极佳 | 社区体量偏小，部分边缘场景适配有限               |

综合权衡后，项目最终采用 **Vite + @crxjs/vite-plugin** 方案。其最大优势是极致的开发体验：修改Popup UI、样式、业务逻辑后，可实时热更新，无需手动刷新扩展、重启浏览器，大幅提升迭代效率。同时，工具可自动同步 `package.json` 版本号至`manifest.json`，省去手动维护版本的重复工作。在当前前端生态下，若无特殊兼容需求，Vite是所有前端新项目的首选构建工具。

### 5.3.3 权限策略：严格落地最小权限原则

权限设计是Chrome扩展开发的核心命脉，直接决定用户信任感、应用审核效率与产品安全性，也是本项目最关键的技术决策之一。

市面上绝大多数翻译扩展都会声明 `<all_urls>` 全局主机权限，允许读取和修改用户所有访问页面的数据。该方案虽能简化开发、实现全页面自动翻译，但存在两大致命短板：一是触发Chrome严苛审核机制，拉长审核周期、提高隐私说明复杂度；二是安装时弹出高危权限警告，大幅降低用户信任度。

本项目全程遵循**最小权限原则**，仅声明三项刚需核心权限：`storage`、`activeTab`、`scripting`，无任何多余授权。

项目采用「按需注入、非常驻运行」的轻量化设计：Content Script 不会随页面自动加载，仅当用户主动点击翻译按钮时，由Popup调用 `chrome.scripting.executeScript` 动态注入当前页面；页面关闭或刷新后，注入脚本自动失效，无后台残留、无持续监听。

该方案牺牲了部分开发便捷性，需要额外实现一套 `ensureContentScript()` 注入检测机制，通过「状态探测、动态注入、多次重试确认」的闭环流程，保障脚本稳定加载。但换来的是极简安全的权限声明、无高危安装警告、快速的商店审核流程，是兼顾安全、体验与合规的最优设计。

在整套技术选型中，Claude Code的核心价值是**客观拆解方案利弊、梳理约束条件、列出决策依据**，而非直接替开发者做决定。技术选型的最终取舍，需要开发者结合项目定位、风险偏好、长期目标自主判断，AI始终是辅助分析工具，而非决策主体。

## 5.4 产出标准化技术Spec：AI编程的核心护栏

需求澄清、技术选型全部落地后，绝对不可以直接写代码。AI工程化的核心铁律是：**先产出完整、严谨、可落地的技术Spec，再启动编码开发**。

在AI编程场景中，AI会忠实执行你的指令，但无法自主预判隐性约束、边界规则、架构规范。如果缺少Spec约束，AI很可能快速写出大量“功能可用、架构跑偏、边界缺失、无法迭代”的代码，最终导致大规模返工。

本项目完整技术规范文档存放路径：`docs/superpowers/specs/2026-05-19-itranslate-design.md`。作为项目第一份正式基准文档，后续所有代码开发、功能迭代、Bug修复、逻辑优化，均严格以此为标准。

这份标准化Spec覆盖项目全维度核心规范，完整结构与核心内容如下：

1. **核心定位**：一句话清晰定义产品形态、目标用户、核心价值与差异化优势，为所有开发决策提供统一锚点；
2. **功能边界**：明确V1版本待实现功能清单，同时定义 **Out of Scope 排除项**，明确不做自动翻译、多语种互译、Firefox适配等功能，杜绝需求蔓延；
3. **技术架构**：定义四大隔离执行上下文（Background / Content Script / Popup / Settings），明确各模块职责边界、依赖关系与通信规则；
4. **消息协议**：规范12种核心Action类型，统一消息流向、Payload数据格式、异常回调与错误处理机制；
5. **完整数据流**：梳理从用户点击按钮、脚本注入、文本提取、过滤处理、接口请求、结果缓存到页面渲染的10步完整执行链路；
6. **错误处理矩阵**：针对未配置API、接口限流429、网络超时、参数异常等场景，制定标准化重试、提示、降级策略；
7. **缓存策略**：基于IndexedDB实现本地缓存，通过djb2哈希+文本长度+目标语言组合生成缓存Key，同时做原文校验，杜绝哈希碰撞；
8. **测试策略**：搭建单元测试、集成测试、E2E端到端测试三层测试体系，明确各层级测试工具、覆盖范围与落地标准。

### 5.4.1 AI编程场景下，Spec的核心价值

传统团队开发中，Spec的核心作用是**团队认知对齐**。而在AI编程中，Spec拥有更关键的价值：**它是给AI的精准需求手册与架构护栏**。

人类同事可以凭借经验主动补全隐性规则、追问模糊点，但AI的执行高度依赖输入信息。一份完整的Spec，会提前把所有架构约束、边界规则、命名规范、错误策略、功能取舍全部固化，让AI无需猜测、不会跑偏、始终按照统一标准开发。

本项目实战充分验证了这一点：定稿Spec之后，后续所有模块开发几乎没有出现任何方向性返工。AI在每一轮编码、重构、优化时，都会严格遵循Spec中的架构约定，例如“Content Script不直接调用API、所有翻译请求必须经过Background转发”等核心规则，全程保持代码架构统一、质量稳定。

可以总结出一条高效AI编程公式：**1小时写Spec，节省10小时返工**。Spec不是形式化文档，而是AI开发的核心风控工具。

## 5.5 核心架构：四大上下文隔离设计

Spec中最核心的设计，是Chrome扩展MV3架构下的**四大执行上下文隔离模型**。整套架构职责清晰、分层明确，完美适配MV3的服务进程约束与隐私权限规范，也是本项目所有代码的底层依据。

```plain
                    chrome.runtime.sendMessage
┌──────────────┐ ◄─────────────────────────► ┌──────────────────┐
│    Popup     │                               │   Background     │
│  工具栏弹窗   │                               │   翻译引擎+缓存    │
│              │                               │                  │
│ 翻译/撤销按钮  │                               │ translateBatch() │
│ 语言选择      │                               │ translateDictionary()  │
│ 划词开关      │                               │ cacheGetBulk()   │
└──────┬───────┘                               └────────┬─────────┘
       │                                                │
       │ chrome.scripting.executeScript                 │ chrome.runtime
       │ (按需注入 content.js)                            │ .sendMessage
       │                                                │
       ▼                                                ▼
┌──────────────────────────────────────────────────────────────────┐
│                        Content Script                           │
│                       (页面内执行)                                │
│                                                                  │
│  extractRawSegments()  ──►  structuredFilter  ──►  renderTranslations() │
│       (遍历 DOM)              (过滤噪音)              (注入翻译)    │
│                                                                  │
│  MutationObserver ──►  catchUpNewContent()  (动态内容追扫)       │
└──────────────────────────────────────────────────────────────────┘
┌──────────────┐
│   Settings   │  (独立选项页，配置 API endpoint / key / model / prompt)
└──────────────┘
```

整套架构包含四大独立模块，各司其职、互不越界，通过标准化通信机制协同工作：

- **Popup（弹窗交互层）**：承载用户所有操作，包括翻译触发、撤销翻译、语言切换、划词开关控制，是用户与扩展的交互入口；
- **Background（后台服务层）**：纯计算与状态管理中心，负责API请求、批量翻译、词典查询、本地缓存读写，统一保管API密钥等敏感数据，不操作任何页面DOM；
- **Content Script（页面执行层）**：运行在网页上下文，负责DOM遍历、文本提取、噪音过滤、翻译结果渲染、动态内容监听，仅处理页面交互，不调用任何外部API；
- **Settings（配置页面）**：独立配置面板，支持用户自定义API地址、密钥、模型类型、翻译Prompt，实现个性化配置。

### 5.5.1 两大核心架构设计决策

**1. 职责彻底隔离：后台不碰DOM，页面不碰接口**

这是MV3架构的核心精髓。Background作为Service Worker无DOM操作权限，专注纯业务计算、数据缓存与接口请求；Content Script拥有完整DOM权限，负责页面交互与视图渲染，但不持有敏感密钥、不直接发起API请求。所有跨模块交互均通过 `chrome.runtime.sendMessage` 通信，职责边界绝对清晰，彻底规避状态混乱、权限泄露问题。

**2. 按需注入：Content Script 非常驻、无全局权限**

配合最小权限策略，项目不全局声明页面注入权限，Content Script不会自动挂载到所有网页。仅当用户主动点击翻译按钮时，由Popup动态注入脚本；页面关闭、刷新后脚本自动销毁，实现“用时激活、用完即止”的轻量化运行模式，兼顾功能体验与用户隐私安全。

## 5.6 本章核心实战技巧

结合完整立项与Spec落地流程，总结五条可直接复用的AI编程立项技巧，帮你规避绝大多数新手坑点：

1. **想法模糊绝不写代码，先用 brainstorming 收敛需求**。不要直接丢一句模糊需求让AI开发，让AI反向提问、补齐盲区。项目定位、用户群体、触发方式、功能边界，全部前置敲定，后续代码质量才有保障。
2. **Spec 必须明确「不做什么」**。只写待实现功能会导致需求持续蔓延，开发中不断新增边界功能、打乱迭代节奏。明确 Out of Scope 排除项，能帮你全程坚守项目边界，避免无限迭代。
3. **技术选型让AI分析、由你决策**。AI擅长罗列利弊、梳理约束、输出客观信息，但无法替代你的价值判断与项目取舍。开发速度、长期维护、审核合规、用户体验的优先级，必须由开发者自己定义。
4. **权限策略一次性做对，绝不后期追加**。Chrome扩展权限具有不可逆性，后期删减权限无法同步覆盖已安装用户，且宽泛权限会拉高审核与信任成本。立项阶段就落地最小权限原则，彻底规避后续隐患。
5. **架构图无需精美，但必须存在**。简单的ASCII架构图足矣，它是你和AI的统一协作语言。任何代码改动前，都可以对照架构图校验是否符合分层规则，避免架构跑偏、职责混乱。

## 5.7 本章小结

本章完整复盘了iTranslate项目从模糊想法到标准化技术Spec的全流程立项逻辑，核心要点汇总如下：

- **定位先行**：借助 brainstorming 结构化提问，将模糊的“翻译扩展想法”，收敛为清晰的“面向开发者、可自定义密钥与Prompt、手动触发的轻量化翻译工具”，为所有开发决策提供核心锚点；
- **选型有理**：全套技术栈选择均基于约束与权衡，Manifest V3保障长期兼容、Vite+crxjs提升开发效率、最小权限策略保障安全与审核效率，无主观随意选型；
- **Spec 是AI编程的核心护栏**：标准化技术Spec包含定位、边界、架构、协议、数据流、报错处理、缓存与测试策略，前置1小时梳理，可规避数十小时的无效返工；
- **架构隔离是项目基石**：四大上下文分层架构，实现职责解耦、权限隔离、通信标准化，适配MV3核心约束，保证项目可维护、可迭代；
- **最小权限是长期优势**：storage + activeTab + scripting 三项最小权限，从立项阶段就守住隐私安全与用户信任底线，规避审核风险。
